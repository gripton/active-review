What can you do on the editor screen?

Select requirements material, choose a reviewer, create session questionnaire and release the session for review.

In the questions edit section:

The editor page allows you to type a question about the software requirement specification into the ARR tool.

Edit Question (edit box)- you use this edit area to insert the questions you have created into the editor screen.
Add Question (button)- when you have completed your work on a specific question, you use this button to actually add the question to the list.

For questions you have added:

Edit- pulls up a editor window with the current question ready for you to modify in the same way you would when you first create the question.
Delete- removes the question and renumbers the remaining ones.

On the upper navigation bar:

Home- returns you to the home page where you can create, open, and view new and existing sessions.
File (menu)- allows you to initiate a number of important actions related to the session you are creating such as- rename, save, preview, release, assign reviewer, spawn. These are described in detail below.
Help on ARR activities- more information and assistance with “selecting requirements”, “selecting the reviewer”, and “creating the questionnaire” (this link)

From the file menu, options are:

Rename (pop up)- changes the title/name of the review session. Allows you to give the review session a descriptive name so it can be distinguished from other sessions.
Save- saves the contents of the session editor screen. To keep the content you have entered you must save before leaving the editor page.
Preview- formats the contents of the session editor screen in the same way the reviewer will see the screen once the session is released to him/her. You can check the presentation of the requirements and questions (along with your comments) in this screen.
Spawn- allows you to copy specified sections of the review session to a new session. This makes it easy for you to replicate a session or reuse parts of an existing session to create a similar session.
Assign reviewer- assigns a reviewer to this session. The session cannot be released until you assign a reviewer. (More information is available in the “selecting the reviewer” section).
Release- moves the review session from the create stage to the release stage so the assigned reviewer can start the review process. After a session is released it can no longer be edited by the creator.

A suggested (ideal) process

Creating the questions for the session's questionnaire

Start with understand of the goals and objectives for the session- what we want to accomplish, what type of errors we want to uncover or what qualities of the requirements we want to validate

Determine the general types of questions and questioning techniques that are useful for accomplishing the objectives of the session

Assure you understand the requirements specification selected for this session

Organize the requirements into reviewable sections- details pertaining to different classes of questions isolated in their own sections to the extent possible

Create a few questions (at most 5-10) that cover one central detail of the session from highest to lowest level of the scope or most general to most specific- this leads to a small “vertical slice” of review questions

Create a few high-level or general questions that cover topics at the boundary of the scope of the session- this leads to a “horizontal slice” (breadthwise) of the review questions

Fill in the details between these boundaries (depth and width) until the coverage of the requirements specified is complete- each question should pertain to or “cover” a goal of the session for a specific area of the specification

Before releasing the session make a final check to confirm that questions cover the entire scope of the review (both in level of detail and breadth of sections of the specification)

Characteristics of review session questions

Questions should have a single “concern” per question- avoid mixed purposes or objectives in a single question

Questions should provide the necessary context- for example when a question measures conformance to a standard, rule, or domain specific convention, it should includes details by which to measure requirement

Questions can ask for concrete use of requirement- for example you can ask the reviewer to explain what an application would do with specified inputs

Questions should lead to descriptive answers rather than simple affirm/deny (“yes/no”) answers- start questions with “Which…” or “What…” rather than “Does…” or “Is…” to yield more open questions

(Meta-questions can determine the effectiveness or quality of the session and the session questionnaire- ask the reviewer to evaluate the session using questions about the questions themselves or about the review session- this discussion can also take place in the forum part of the ARR tool)

Methods for examining requirements with questions

Ask reviewer to use the requirements to create a draft implementation of one feature (a possible architectural view or summary design) to determine if requirements are complete or clear enough to be useful for downstream use

Ask reviewer to restate a requirement to determine how it will be interpreted by others

Ask reviewer to summarize the content or point of a section or group of related requirements to determine if organization is clear and interpretation is in line with the creators intent

Ask reviewer to point out what parts of the requirements specification he/she would use to answer a specific question about a specific behavior or quality of the system

Ask the reviewer to determine how the system should react to a specific environment or input condition- the condition could be normal or abnormal

Use a requirement testing methodology described below

Requirements “testing” method (adapted from Gause and Weinberg, Exploring Requirements, Quality before Design)

Use requirements that describe system’s externally visible behavior as the basis for the session

The requirements creator should write requirements so they answer questions about how this simulated system behaves

Ask reviewer to simulate the behavior of a system that would fulfill these requirements

Use series of “what if” questions to determine normal use, abnormal but reasonable use, abnormal and unreasonable use

Document test cases and the outcomes when the reviewer “runs” them
